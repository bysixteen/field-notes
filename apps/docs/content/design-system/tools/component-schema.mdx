title: Component contract schema
description: Canonical schema for per-component contracts — a single Markdown sidecar colocated with the component, with four required sections and two recommended sections gated by trigger conditions.
---

A component contract is the artifact that tells humans, agents, and tooling how a single design-system component is meant to be used. This page settles the format question and defines the canonical schema.

## Decision

Every component ships **one artifact**: a Markdown sidecar.

| File | Location | Authority |
|------|----------|-----------|
| `<Component>.contract.md` | Colocated with `<Component>.tsx` | Canonical. Hand-authored when scaffolding; updated when the component evolves. |

There is no JSON sibling. Markdown is the source of truth.

## Why one Markdown sidecar

**Colocation over centralization.** The contract lives next to the code it describes. A reviewer reading a PR sees the contract change in the same diff as the implementation change. A central registry (one big JSON file at the repo root) drifts from the code it nominally describes within weeks of contact with practice.

**Markdown over JSON.** A project-agnostic framework has to optimise for the most common operations: reading, reviewing in PRs, hand-editing during scaffolding. Markdown wins all three. Agents parse Markdown well enough — the headings and tables in this schema are predictable enough to pattern-match against. The token-efficiency argument for JSON only matters when the contract is large and read repeatedly by agents in tight loops; for per-component contracts (4–7 sections, read on demand), it does not pay back the friction it imposes on humans.

**One artifact, not two.** A two-file shape (JSON + narrative Markdown) creates a synchronisation problem. Either the JSON drifts from the prose, or the prose drifts from the JSON, or both drift from the `.tsx`. Collapsing to one file removes the problem entirely. The `.tsx` signature stays the ultimate authority on props; the contract mirrors it.

## Required sections

Every contract has these four sections. They cover the essential job: tell humans and agents what the component is and how to wire it.

### `## Props`

Markdown table: prop / type / default / notes. Lift names and types directly from the `.tsx` signature — the TypeScript types are authoritative. Each row records the prop name, its type, default value (if any), and a short note (optional, required, or a one-line clarification).

```md
## Props

| Prop | Type | Default | Notes |
|------|------|---------|-------|
| state | ButtonState | rest | rest \| hover \| active \| selected \| disabled \| resolving \| pending |
| emphasis | ButtonEmphasis | high | high \| medium \| low |
| sentiment | ButtonSentiment | neutral | neutral \| warning \| highlight \| new \| positive |
| size | ButtonSize | md | xs \| sm \| md \| lg \| xl |
| leadingIcon | React.ReactNode | — | optional |
| trailingIcon | React.ReactNode | — | optional |
| onClick | (e: React.MouseEvent) => void | — | optional |
| children | React.ReactNode | — | required |
```

### `## Dimension encoding`

Markdown table: dimension / `data-*` attribute / values / default. Documents how the component projects the [Dimensional Model](/design-system/model) onto DOM attributes. CSS selectors target these attributes; this table is the contract between the prop API and the styling layer.

```md
## Dimension encoding

| Dimension | data-* attribute | Values | Default |
|-----------|------------------|--------|---------|
| state | data-state | rest, hover, active, selected, disabled, resolving, pending | rest |
| emphasis | data-emphasis | high, medium, low | high |
| sentiment | data-sentiment | neutral, warning, highlight, new, positive | neutral |
| size | data-size | xs, sm, md, lg, xl | md |
```

A component does not need to participate in every dimension. Badge skips `state` and `emphasis`; Alert keeps only `sentiment`. Omit rows for dimensions the component does not use.

### `## Token bindings`

Bullet list of `--var-name → semantic role`. Records which CSS custom properties the component reads, paired with the role each plays in the component's visual surface. This is the contract between the component and the [token chain](/design-system/token-chain).

```md
## Token bindings

- `--button-bg` → background fill
- `--button-fg` → label and icon colour
- `--button-border` → border colour at every emphasis level
- `--button-radius` → corner radius
- `--button-padding-x` → horizontal inset
- `--button-padding-y` → vertical inset
- `--button-height` → resolved height per size step
```

Use semantic role names, not hex values or token paths. The role names map to the [DESIGN.md surface vocabulary](/design-system/manifest-architecture).

### `## Usage rules`

Prose: when to use, anti-patterns, primitive base. Three short paragraphs are usually enough.

- **When to use.** The decision boundary against neighbouring components — what makes this primitive the right pick over its siblings.
- **Anti-patterns.** Three to five concrete misuses, phrased as imperatives ("do not pass a `loading` boolean — use `state='resolving'` instead"). Keep the list short; each entry should map to a real misuse seen in the wild.
- **Primitive base.** The underlying element or library — `native <button>`, `Radix Dialog`, `Radix Popover`. Critical for understanding inherited semantics.

```md
## Usage rules

Use Button for actions that change application state. For navigation between pages or routes, use Link. For toggle-style controls, use ToggleGroup.

Do not pass a `loading` boolean — use `state='resolving'` instead. Do not introduce `isPrimary` / `isGhost` booleans — emphasis is an enum. Do not render a Button without a label or aria-label. Do not stack two high-emphasis Buttons in the same action group.

Built on the native `<button>` element. Inherits implicit role, keyboard activation (Enter, Space), and form-association behaviour.
```

## Recommended sections

Add these only when the component warrants them. They are not required for every component.

### `## A11y`

Keyboard interactions, ARIA roles, screen-reader behaviour.

**Add when:** the component has a Radix base, manages focus, or implements non-trivial keyboard interaction (arrow-key navigation, Escape-to-dismiss, type-ahead).

**Skip when:** the component is a static-display primitive (Badge, Card, Avatar) whose accessibility is fully covered by the underlying element. Coverage in those cases lives in Storybook a11y tests and the project's accessibility audit pipeline — not in the contract.

### `## Behavior`

State transitions, focus management, async behaviour.

**Add when:** the component manages internal state (open/closed, hover-with-delay, resolving with optimistic UI), traps focus, or has async transitions worth pinning down.

**Skip when:** the component is stateless and its rendering is fully determined by its props.

The trigger for A11y and Behavior is the same: there is interactive complexity that prop documentation alone cannot capture. If the component is a primitive that renders, the contract is fully described by the four required sections.

## Deferred section

Add per-component only when real drift surfaces — not as a default.

### `## Execution rules`

Vocabulary for strict component behaviour: `layoutEffect`, `allowedMechanisms`, `forbiddenMechanisms`, `paddingCompensation`, `interactivity`, `widthBehaviour`, `heightBehaviour`. Captures the constraints that govern *how* a component is implemented, not just *what* it does.

**Add when:** a real drift case has surfaced in review or production — a component is being reimplemented in a way that violates an unwritten constraint, and the contract needs to make that constraint enforceable.

Until then, do not preemptively add this section. Foundation-level defaults for execution rules stay parked until enough component-level contracts exist to extract meaningful patterns.

## Authority and authoring workflow

1. **Start from the `.tsx` signature.** The TypeScript types are authoritative for `## Props`.
2. **Hand-author the contract when scaffolding.** A new component ships its `.contract.md` on day one — this is the moment when the API and intent are clearest.
3. **Update the contract when the component evolves.** Every PR that changes the prop surface, dimension encoding, or token bindings updates the contract in the same commit.
4. **Reviewers read the contract first.** When the contract diff and the implementation diff sit side by side in a PR, drift is caught at the moment it would otherwise enter the codebase.

## Why JSON-first was wrong

This page previously specified a 7-field JSON canonical (`name`, `anatomy`, `dimensions`, `props`, `a11y`, `behavior`, `doNot`, `pairedComponents`) with Markdown as a generated narrative layer. That decision is deprecated.

**What changed:** zero components were ever authored against the JSON-first shape. In practice, contracts that *do* exist in consumer projects are hand-authored Markdown — and they ship without the synchronisation overhead a two-file shape would have imposed. The JSON-first decision did not survive contact with practice.

**The token-efficiency argument did not pay back its cost.** The original justification was that agents read structured data more cheaply as JSON than as Markdown. True in the limit; not true at the scale of per-component contracts (4–7 sections, read on demand) where the human-authoring cost dominates. Removing the JSON sibling removes the synchronisation problem and costs nothing meaningful at parse time.

**The two-file shape was the real friction.** Even if JSON had been the better machine surface, maintaining a JSON sibling alongside a Markdown narrative meant every contract change happened in two places. One source of truth, in the format humans review most easily, is the right primitive.
