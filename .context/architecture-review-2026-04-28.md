## Section 2 — Fit-for-purpose verdict

**`component-schema.mdx` (7-field JSON):** **Rebuild.** The "JSON is the source of truth" decision introduces unnecessary friction for human authoring and review. A project-agnostic framework should prioritize readability and maintainability. A Markdown-first approach with a single sidecar file per component provides a better developer experience and is equally parseable by agents. Verdict: rewrite to document a Markdown-first sidecar shape; deprecate the JSON-first decision.

**Phase 6 PR1 (execution-rules vocabulary):** **Rewrite scope.** The vocabulary (`allowedMechanisms`, `forbiddenMechanisms`, `layoutEffect`, etc.) is useful for defining strict component behaviors. However, embedding it in `components.json` as `behaviorContract` centralizes data that should be colocated with the component. The vocabulary stays available as an OPTIONAL `## Execution rules` section added to the component sidecar when strict behavior definition is needed. The foundation-level defaults (`EXECUTION-CONTRACT-DEFAULTS.md`) stay parked until sufficient component-level contracts exist to extract meaningful defaults. Verdict: don't ship the foundation file; document the vocabulary as an optional sidecar section.

**`ARCHITECTURE.md` (two-file split):** **Rewrite.** The `contract.md` + `components/<name>.md` split is overly complex. It collapses into one file: a `<Component>.contract.md` sidecar. The `fn-contract` skill (separate from `fn-component-md`) is redundant — it collapses into the per-component sidecar generator. Verdict: rewrite to align with a single-file shape; drop `fn-contract`; rename `fn-component-md` to `fn-component-contract`.

## Section 3 — Recommended single direction

**Adopt a single, Markdown-first `.contract.md` sidecar per component.**

One file per component, colocated with the `.tsx` source. Markdown is the primary artifact. Four core sections: Props, Dimension encoding, Token bindings, Usage rules. Hand-authored or template-generated when scaffolding a new component; updated when the component evolves.

This dissolves both Phase 6 PR1 (no `behaviorContract` in `components.json`) and ARCHITECTURE.md's two-file split (no separate `contract.md` + `components/<name>.md`). It provides a clean, project-agnostic primitive that works for both human developers and AI agents.

**Why this shape is the right primitive:**

1. It colocates documentation with the code it describes.
2. Its 4 core sections cover the essential job: telling humans and agents what the component is and how to wire it.
3. Markdown is universally readable, diffable in PRs, and easily parsed by LLMs without the strict formatting overhead of JSON.
4. The execution-rules vocabulary (Phase 6 PR1) can easily be added as an OPTIONAL section when a specific component requires strict behavior definition.
5. It avoids the complexity of synchronizing multiple files per component.

**What this means for each artifact:**

- `component-schema.mdx`: rewrite to document the single Markdown sidecar shape; deprecate JSON-first decision.
- `components.json` at root: strip token-slot wiring; become a thin primitive index (name → sidecar path). Remove pseudo-primitive state variants.
- `ARCHITECTURE.md`: rewrite to reflect the one-file shape; drop `fn-contract` skill; correct factual inaccuracies.
- `PHASE-6-EXECUTION-CONTRACTS.md`: rewrite to document `## Execution rules` as an optional per-component section; park foundation defaults.
- `bs-design-md`: reconcile the schema mismatch — live `components.json` uses token-slot wiring, not the `applies_to`-shaped schema the `--from-dimensional` walker expects. Fix the live file as part of the `components.json` rebuild (#170).
- `bs-init`: add inline `generate-ramps.ts` invocation; seed sidecar templates for the default primitives; add `--prompt` mode.
- `fn-component-contract` (new skill): generates or updates the per-component sidecar from `.tsx` + `tokens.json` + `DESIGN.md`.

## Section 6 — Open decisions (judgment-not-evidence)

1. **Required vs. Recommended Sections — RESOLVED.** `## A11y` and `## Behavior` are **recommended, not required.** Add when the component has a Radix base, focus management, or non-trivial keyboard interaction. Static-display primitives do not carry these sections. Coverage lives in Storybook a11y tests and `bs-accessibility` audits, not in the sidecar by default. The empirical signal is 14 shipped contracts that omitted both sections without producing a11y regressions — codify the practice rather than retrofit.

2. **Generator vs. guided template for `fn-component-contract` — RESOLVED.** **Guided template first.** Ship 5+ hand-authored contracts. Then assess what is repetitive enough to warrant a generator and what is judgment-laden enough that a generator would hallucinate. Promote to generator only if the repetitive surface is real. The JSON-first decision in `component-schema.mdx` did not survive contact with practice (#166 exists because of it); the conservative move is correct here. (→ issue #169 updated)

3. **Sidecar location:** Alongside the `.tsx` (colocated) vs. a parallel `contracts/` folder (cleaner separation, breaks colocation). Lean toward colocated but worth a confirm.

4. **bs→fn rename timing relative to schema rewrite:** Parallel (faster) vs. sequential (cleaner blame). Lean parallel.

## Section 8 — Per-question answers

**Q1: Is ARCHITECTURE.md's proposed two-file contract shape optimal?**
No. ARCHITECTURE.md proposes two files (`contract.md` + `components/<name>.md`). A single, colocated `<Component>.contract.md` sidecar is simpler, easier to maintain, and avoids synchronization issues. The two-file split collapses into one.

**Q2: Is component-schema.mdx's 7-field JSON shape the right canonical source?**
No. A Markdown-first approach is more accessible for human developers and equally parseable by agents. The JSON-first assertion adds unnecessary friction.

**Q3: Is Phase 6 PR1's `behaviorContract` embedding in components.json correct?**
No. Centralizing behavior contracts in `components.json` separates documentation from the component. The execution-rules vocabulary belongs as an optional sidecar section in the component's directory.

**Q4: Is `EXECUTION-CONTRACT-DEFAULTS.md` needed?**
Not yet. Park until sufficient component-level contracts exist to warrant extracting foundation defaults.

**Q5: Does `bs-init` invoke `generate-ramps.ts`?**
No. `bs-init/scripts/init.mjs` uses `copyDir` from templates only. This is an inaccuracy in ARCHITECTURE.md.

**Q6: Does `bs-design-md --from-dimensional` work with the live `components.json`?**
No. The `--from-dimensional` walker (`dimensional-walker.mjs`) is implemented and functional. The live `components.json` uses token-slot wiring without `applies_to` — the schema the walker expects. The mismatch is in the data, not the code. Fix: rebuild `components.json` as a thin index with `applies_to` (→ issue #170).

**Q7: Does Phase 6 PR1 conflict with ARCHITECTURE.md?**
The conflict dissolves when adopting a single Markdown sidecar. Phase 6 PR1's `behaviorContract` embedding and ARCHITECTURE.md's two-file split both collapse into one file: the per-component `.contract.md` sidecar.

**Q8: What is the right sequencing for the rename and the schema rebuild?**
bs→fn rename can run in parallel with the schema rewrite (Decision PR 1) if isolated to skill files. The rename is mechanical. See Section 5 for full sequencing.
