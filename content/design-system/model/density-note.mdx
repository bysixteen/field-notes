---
title: Density — why it is not a dimension
description: Density was considered as a sixth dimension in the bysixteen model. This note documents why it was not adopted and how Size covers the same use cases.
---

## What Density would have been

Density was proposed as a sixth dimension to allow compact vs. default spacing without changing the Size dimension. The intent was that `data-density="compact"` would reduce internal padding and gap values across components, enabling denser information layouts (for example, admin data tables) without changing the semantic size of the component.

## Why Size covers this

A smaller `data-size` value achieves the same result. The Size dimension already encodes how much space a component occupies. Using a smaller `data-size` for compact layouts is semantically correct and avoids introducing a dimension whose only role is to modulate spacing.

Introducing Density would also create ambiguity: a component with `data-size="sm" data-density="compact"` has two attributes competing to control the same visual property. Size alone is unambiguous.

## The decision

**Density is not a dimension in the bysixteen model.**

The canonical five dimensions are:

| Dimension | Role |
|-----------|------|
| Sentiment | Communicative intent (`neutral`, `highlight`, `positive`, `warning`, `new`) |
| Emphasis | Visual weight (`low`, `medium`, `high`) |
| Size | Spatial footprint — also covers compact layouts via smaller values |
| Structure | Component-specific anatomy (e.g. `standard`, `text`); never exposed as a runtime prop |
| State | Interaction state (`rest`, `hover`, `active`, `selected`, `disabled`, `resolving`, `pending`) |
