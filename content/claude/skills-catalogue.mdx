Skills are reusable instruction sets that Claude follows for specific workflows. Instead of re-explaining the same process each conversation, you invoke a skill and the agent follows a tested, consistent playbook.

This page catalogues the **field-notes toolkit** — the skills bundled in `dist/field-notes-toolkit.plugin`. For how skills work in general and how to create them, see [Skill-Based Automation](/claude/skills-patterns).

For agent context, see also [DESIGN.md and the Dimensional Model](/claude/dimensional-and-design-md).

## The `_foundations/` layer

Every `bs-*` skill leans on a shared mental-model layer in `.claude/skills/_foundations/`. Five files:

- **`DESIGN-INTENT.md`** — what the toolkit is for and the assumptions every skill is built on.
- **`DIMENSIONAL-MODEL.md`** — the Sentiment × Emphasis × Size × Structure × State vocabulary, the canonical value lists, and the decision order.
- **`TOKEN-ARCHITECTURE.md`** — the four-layer cascade (primitive → semantic → component → variant) and how aliases flow through it.
- **`QUALITY-GATES.md`** — severity model and verdict logic shared by `bs-review`, `bs-accessibility`, and the audit skills.
- **`SELF-REVIEW.md`** — the four-step protocol every skill calls before declaring done (re-read brief → list assumptions → list what wasn't checked → green-light gate).

If a skill references "the dimensional model" or "the cascade", it points here.

## Available Skills

### Specification

#### bs-design-md

**Invoke:** `/bs-design-md` | **Status:** Available

Projects a project's tokens and components into the spec-compliant Google `DESIGN.md` format with paired DTCG `tokens.json` sidecar. The closer for the design loop — three modes:

- **Extract mode** — point at a live site URL, get a `DESIGN.md` plus DTCG `tokens.json`. Bootstraps a new product from a reference.
- **Project mode** (`--tokens` + `--components`) — take an existing DTCG `tokens.json` plus a hand-authored `components.json`, emit the spec-compliant pair.
- **Dimensional mode** (`--from-dimensional <root>`) — walk the consumer's `tokens.json`, `components.json` (with `applies_to`), and the five model MDX files, then synthesise the flat per-variant components block.

Outputs include preservation markers (`<!-- bs-design-md:generated:start --> … :end -->`) so consumer-authored prose below `:end` survives regenerations.

`bs-design` is a one-line redirect stub to this skill.

---

#### bs-tokens

**Invoke:** `/bs-tokens` | **Status:** Available

Consume, audit, and reason about design tokens. Covers severity tiers (S0–S3), DTCG conformance, alias-chain integrity, and the Token Trap discipline (never load whole token system into context — query specific tokens via the manifest).

---

### Bootstrap

#### bs-init

**Invoke:** `bs-init <name> <dir>` | **Status:** Available

Scaffolds a fresh project's canonical four-piece context stack so it can immediately consume `bs-design-md --from-dimensional`:

| Artefact | Purpose |
|----------|---------|
| `tokens.json` | DTCG 2025.10 token source. Six placeholder anchor colours. Replace with real values. |
| `components.json` | Five canonical primitives (`button`, `input`, `badge`, `alert`, `card`) declared with `applies_to: all`. |
| `content/design-system/model/{sentiment,emphasis,size,state,structure}.mdx` | The five-dimensional vocabulary, each with `dimensional_values: { default, values }` frontmatter. |
| `CLAUDE.md` | Starter system-prompt pointing at the field-notes toolkit and `llms.txt`. |
| `DESIGN.md` | Empty preservation markers ready for the first `bs-design-md` emit. |

After `bs-init`, run `bs-design-md --from-dimensional <root>` against the same directory and the scaffold produces a valid `DESIGN.md`.

### Component creation

#### bs-component-api

**Invoke:** `/bs-component-api` | **Status:** Available

Defines a component's consumer-facing contract — props, types, defaults, slots. Enforces the four-input categories rule (Dimension / Content / Slot / Event), the enums-not-booleans rule, the every-input-has-a-default rule, and the dimension-not-visual naming rule.

Audit checklist included for existing component APIs.

---

#### bs-component-scaffold

**Invoke:** `/bs-component-scaffold` | **Status:** Available

Generates the standard six files from a `bs-component-api` output: `component.tsx`, `component.types.ts`, `component.css` (or `.scss`), `component.stories.tsx`, `component.test.tsx`, `index.ts`.

---

#### bs-css

**Invoke:** `/bs-css` | **Status:** Available

Audits component CSS against the token system. Flags hardcoded values, broken alias chains, missing state coverage, and wrong cascade direction.

---

#### bs-html

**Invoke:** `/bs-html` | **Status:** Available

Decides element semantics and ARIA for a component. Maps component types to the correct HTML primitive, role, and key ARIA attributes (`button` / `link` / `dialog` / `tab` / `progressbar` / etc.).

---

### Review

#### bs-review

**Invoke:** `/bs-review` | **Status:** Available

Orchestrates a six-stage scored component review across `bs-component-api`, `bs-react-patterns`, `bs-html`, `bs-accessibility`, `bs-tokens`, and `bs-css`. Produces a structured findings file with severities and a verdict (ship / fix / block).

---

#### bs-accessibility

**Invoke:** `/bs-accessibility` | **Status:** Available

WCAG 2.2 AA audit framework structured around the bracket model (B1 Display / B2 Interactive / B3 Form / B4 Composite / B5 Data / B6 Overlay). Returns a checklist of violations with severities and remediation notes.

---

#### bs-react-patterns

**Invoke:** `/bs-react-patterns` | **Status:** Available

Reviews component code against modern React idioms — hook rules, prop drilling, render-time vs effect-time work, suspense boundaries, ref forwarding.

---

#### bs-testing

**Invoke:** `/bs-testing` | **Status:** Available

The unit + accessibility + visual-regression triad. Provides templates for `@testing-library/react` unit tests, `jest-axe` / `@axe-core/playwright` automation, and Storybook+Playwright snapshot loops.

---

### Storybook documentation

#### bs-storybook-ds

**Invoke:** `/bs-storybook-ds` | **Status:** Available

Orchestrates a complete Storybook design-system documentation pass — calls `bs-storybook-foundations` then `bs-storybook-docs` for each component, using `bs-storybook-helpers` patterns throughout.

---

#### bs-storybook-foundations

**Invoke:** `/bs-storybook-foundations` | **Status:** Available

Documents primitives (colours, typography, spacing, radius, shadows) on Storybook foundation pages.

---

#### bs-storybook-docs

**Invoke:** `/bs-storybook-docs` | **Status:** Available

Documents an individual component on a Storybook docs page — anatomy, prop table, do/don't, usage examples, all dimension permutations.

---

#### bs-storybook-helpers

**Invoke:** `/bs-storybook-helpers` | **Status:** Available

The catalogue of helper components used across `bs-storybook-foundations` and `bs-storybook-docs`: `DocPage`, `TokenTable`, `Swatch`, `PropsTable`, `Anatomy`, `DoAndDont`, etc.

---

### Workflow

#### start-issue

**Invoke:** `/start-issue` | **Status:** Available

Bootstraps work on an existing GitHub issue. Creates a correctly named branch, plans the approach, implements changes, verifies the build, and opens a PR against `main`.

**What it asks you:** the issue number or URL.

---

#### start-work

**Invoke:** `/start-work` | **Status:** Available

Begins a new piece of work that does not have an existing GitHub issue. Captures context as a written brief, sets up the branch, and guides commits — does NOT open a PR.

**What it asks you:** what you are trying to accomplish, the scope, and any constraints.

## Installing skills

Skills ship in two distribution paths, mirroring the toolkit shape:

<Steps>

### Plugin installation (recommended)

The `dist/field-notes-toolkit.plugin` artefact bundles every `bs-*` skill, the `_foundations/` layer, and the `start-issue` / `start-work` workflow skills. When installed as a Claude Code plugin, skills live under `.claude/skills/{skill-name}/SKILL.md` inside the plugin directory.

Build the plugin: `node scripts/pack-plugin.mjs` — outputs `dist/field-notes-toolkit.plugin`.

### Direct copy

For a project not using the plugin format, copy the relevant skill directories from `.claude/skills/` directly into the consumer's `.claude/skills/`. The `_foundations/` directory must travel with any `bs-*` skill — the skills resolve cross-references against it.

### Invoke the skill

Type `/{skill-name}` in a Claude Code conversation. The agent reads the skill file, opens any `_foundations/` references, and follows the playbook.

</Steps>
