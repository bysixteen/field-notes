Project Squad is the installable, portable version of the [Pioneer Framework](/principles/pioneer-framework). Where the Pioneer Framework documents the methodology, Project Squad provides the commands, personas, and templates to run it. One command bootstraps the entire operating system into any project.

## Quick Start

```bash
npx project-squad init
```

This creates the directory structure, copies the nine persona definitions, installs the Claude commands, and scaffolds empty living documents. Before running init, create `_meta/PROJECT_CONTEXT.md` describing your project — the init command reads this to seed context.

## What It Creates

```
_meta/
└── PROJECT_CONTEXT.md            # Your project seed (written before init)

.squad/
└── project-squad.md              # Portable: 9 persona definitions (never modified)

.claude/commands/
├── init-project-squad.md         # Scaffolding command
├── create-sprint.md              # Sprint command
├── create-workshop.md            # Workshop command
├── create-spike.md               # Spike command
└── import-personas.md            # Persona import command

research/                         # Project-specific context
├── PRINCIPLES.md                 # Design & technical principles
├── PERSONAS.md                   # Project user personas
├── DECISIONS.md                  # Decision log
├── sprint-status.md              # Sprint history
├── sprint-backlog.md             # Upcoming sprint candidates
├── dissent-register.md           # Dissent log with review triggers
├── sprints/
├── workshops/
└── spikes/

docs/
└── decisions/                    # Architecture Decision Records
```

## Two-Layer Architecture

| Layer | Contents | Changes Between Projects? |
|---|---|---|
| **Layer 1: Portable Toolkit** | `.squad/` personas + `.claude/commands/` | No — travels unchanged |
| **Layer 2: Project Context** | `research/` living documents + `docs/decisions/` | Yes — populated by sprints |

Layer 1 is the constant. It comes from the NPM package and should never be modified per-project. Layer 2 is the variable — it starts as empty templates and grows through sprint work into the single source of truth for the project.

## The Nine Personas

These are archetypes, not real people. Each brings a distinct lens to prevent groupthink and surface blind spots.

| # | Name | Role | Signature Question |
|---|------|------|-------------------|
| 1 | Leo Finch | Visual Designer | "Does this feel like us?" |

| 4 | Kira Sharma | Developer | "What does the implementation actually look like?" |
| 5 | Dr. Aris Thorne | Strategist | "What is the real problem we are trying to solve?" |
| 6 | Rowan Vale | Craftsman | "What is the feeling we want to create?" |

Elias Vance is the **Mandatory Challenger**. His dissent must always be recorded in the Decide phase, even if the team overrules him. His objections feed the dissent register for future review.

## Commands

| Command | When to Use |
|---------|-------------|
| `/create-sprint` | Complex problems needing deep exploration. Supports full 4-phase (Map, Sketch, Decide, Synthesise) or Lite 2-phase mode. |
| `/create-workshop` | Time-pressured decisions requiring a compressed 2-3 hour session. |
| `/create-spike` | Uncertainty is blocking a decision. Must pass the Spike Qualification Test: Can you estimate the effort? Is the goal knowledge, not a feature? |
| `/import-personas` | Bring externally-created client personas into the living `PERSONAS.md` document. |

## Living Documents

Every sprint updates three living documents at the project root:

- **DECISIONS.md** — Append-only decision log with context, alternatives, consequences, and dissent
- **PERSONAS.md** — User personas with confidence levels, updated as understanding grows
- **PRINCIPLES.md** — Design and engineering principles with status (Proposed, Validated, Retired)

These documents are the synthesized output of all project work. They eliminate context burn — new team members or AI agents read these instead of re-reading raw sprint artifacts.

## Relationship to the Pioneer Framework

The [Pioneer Framework](/principles/pioneer-framework) documents the complete methodology: the audit, repository architecture, sprint model, Tiger Team composition, and journey mapping. Project Squad implements the operational parts as portable, installable tooling.

| Aspect | Pioneer Framework | Project Squad |
|---|---|---|
| **Nature** | Methodology documentation | CLI toolkit |
| **Scope** | Full lifecycle (audit → delivery) | Sprint/workshop/spike execution |
| **Team model** | Tiger Team (7 roles) | Nine Personas (archetypal characters) |
| **Installation** | Read and apply manually | `npx project-squad init` |
| **Living documents** | Described in framework | Scaffolded automatically |
